## RWD (Responsive Web Design)란 무엇인가 ?

Responsive Web Design은  

> 하나의 코드 베이스가 모든 화면에 적응된다  

해당 개념이 핵심이다.

이전에는 웹 페이지에서 **PC버전에서 보기/모바일버전에서 보기**와 같은  
웹 뷰 전환 트리거들을 통해 내 디바이스에 적합한 뷰잉 경험을 선택할 수 있었다.

이러한 방식들은 웹과 모바일 사이트를 각각 구축해두고 이를 스위칭하는 개념이다.  

RWD는 이러한 방식보다는 같은 HTML이라면 CSS 코드만으로 화면을 재배치되도록 한 것이다.

## RWD 동작을 위한 핵심 개념

1. Fluid Grid  
기존에 대부분 사용한 **PX** 단위 대신 **%**, **fr**, **flex**와 같은 비율 처리 단위를 사용해서 레이아웃 설정

2. Flexible Media  
이미지나 영상 컨텐츠가 컨테이너에 맞게 늘어나거나 줄어들도록 처리

3. Media Queries  
디바이스 규격에 맞게 스타일 코드를 분리해서 관리하는 방식

세 가지 핵심 내용을 기반으로 사이트를 구축하면 거의 모든 디바이스 환경에서 원활한 UX를 제공할 수 있다.

> 사실 미디어쿼리 없이도 Flex, Grid 설정을 통해 유동적인 레이아웃을 만들고  
> 해당 레이아웃 내부에서 비율을 조작하여 반응형 웹을 구축하는 방식으로 보편화되었다.

### Viewport 이해

구축중인 웹 서비스 코드베이스를 관심있게 보면 `index.html`에서  

```html
 <meta name="viewport" content="width=device-width, initial-scale=1.0" />
```

> width=device-width :뷰포트 폭을 실제 디바이스 폭으로 설정  
> initial-scale=1    :초기 줌 1배  
> maximum-scale=1    :줌 막기 (접근성상 권장하지 않음)  
> viewport-fit=cover :노치 영역까지 콘텐츠 영역으로 사용 (safe-area-inset과 짝)

이런 설정을 확인할 수 있다.  

반응형 처리를 위해 무조건 붙여둬야하는 설정이다.

해당 설정이 의미하는 바는 현재 사용하는 디바이스 폭을 viewport 폭으로 사용하도록 하는 것이다.

## Mobile First

초기에 RWD에 대한 나의 생각은,

> 무조건 모바일 고려하여 먼저 스타일링 하는게 좋다. 데스크톱 먼저 해두고 모바일 작업을 추가하면 리소스가 더 많이 든다.

이런 생각을 가지고 서비스를 개발해왔다.

### Desktop First 문제 경험

- Header UI  
항상 데스크톱 환경에서 헤더 UI를 구축해두면 로고와 네비게이션, 유저 인터렉션 버튼 등 모두 구성해두면 모바일 전환 시
이런 UI들을 재정렬하는 기획과 리소스 분석 시간이 더 오래걸림

- margin / padding  
큰 화면을 기준으로 margin과 padding 등으로 여백/공간을 조절해두면 모바일로 전환 시 이런 공간 관리 요소들을 전부 제거하거나 조절해야하는
어려움이 있음

- User interaction  
데스크톱 환경에서는 마우스를 기본 인터렉션 매개체로 사용하기에 이벤트 관리가 대게 마우스 동작이고 모바일로 전환시 마우스 관련 이벤트 동작 경험이
미스매치한 부분들이 많음

### UX 고려한 설계 방향성 설정

현시점에서 데스크톱 환경보단 모바일에서 웹을 접근하는 사용자가 더 많아졌기에 모바일 트래픽이 더 비중이 큰건 팩트이다.  
그렇기에 모바일에서는 미디어 쿼리 처리를 안 거쳐도 되기에 성능 우위가 있고, 점진적 향상(PE) 철학에도 부합하다.
  
다만 관리 목적으로 사용하는 어드민 페이지의 경우 모바일 보단 데스크톱 사용이 일반적이기 때문에
Mobile First 설계가 오버 엔지니어링이 되는 경우도 있다.

최근들어서는 무작정 `모바일부터 해야돼!` 보다는 `사용성과 상황에 맞게 해야돼!`에 초점을 두고 서비스를 개발 중이다.

## Useful Media Queries

기존 디바이스 폭 관련 처리에만 사용하던 미디어 쿼리 외의 알아두면 유용한 미디어쿼리 문법들을 알아보자

```css
/* 가로/세로 모드 */
@media (orientation: landscape) { ... }

/* 다크모드 */
@media (prefers-color-scheme: dark) { ... }

/* 모션 줄이기 선호 */
@media (prefers-reduced-motion: reduce) { ... }

/* 호버 가능한 입력 장치 (마우스) 여부 — 모바일/터치 구분의 핵심 */
@media (hover: hover) { ... }

/* 화면 픽셀 밀도 */
@media (min-resolution: 2dppx) { ... }
```

내가 학습하며 가장 예의 주시 한 것은 `@media (hover: hover) { }` 이 녀석이다.  

폴더블 모바일 기기나 터치 노트북 등 어떤 디바이스 성격에 국한되지 않고  
다양한 사용자 경험을 제공하는 기기들이 많이 등장하고 있는 세상이다.  
  
단순히 폭을 기준으로 기기를 정의하는 시대는 지나가고 있다고 보고 설계의 방향성도 변화하는 분기점에 있다고 생각한다.

## RWD에서 중요한 CSS 단위에 대하여 알아보자

이 부분 역시 완전 초기 CSS 학습 당시에는

> px 쓰지말고 그냥 rem 써!  
  
이렇게 배웠고, `왜?` 라는 의문을 가지게 되면서 스스로 학습을 많이 했던 부분이다.  

> px은 디바이스가 작던 크건 상관없이 내가 지정한 폭만큼 무조건 화면에 표시된다.  
> em은 내가 속해있는 부모에 의존하여 변화한다. 부모의 폰트크기나 폭을 기준점으로 잡는다.  
> rem은 html 루트에서 정의한 폰트 크기나 폭을 기준으로 잡는다.
>
> 따라서 em의 누적관리 복잡성보다 안전한 rem을 사용하는 것이 가장 이상적이다.
  
### 종류와 쓰임새에 대해 더 자세히 알아보자

절대 단위

- px: CSS에서는 거의 고정 단위로 취급.

상대 단위 — 부모 기준

- %: 부모의 해당 속성 기준.
- em: 부모의 폰트 크기 기준. 폰트뿐 아니라 padding/margin에도 쓸 수 있음. 중첩되면 누적되어 복잡해짐.

상대 단위 — 루트 기준

- rem: 루트(html)의 폰트 크기 기준. em의 누적 문제가 없어 가장 안전. 폰트 크기에 강추.

상대 단위 — 뷰포트 기준

- vw / vh: 뷰포트 폭/높이의 1%.
- vmin / vmax: 가로세로 중 작은/큰 쪽의 1%.
- dvh / svh / lvh: 동적/작은/큰 뷰포트 높이 (모바일 주소창 대응).

권장 사용 방식은  

```markdown
폰트 크기: `rem`,`clamp`  
내부 공간 관리(margin, padding): `rem`,`px`  
컨테이너 폭 관리: `%`,`fr`,`flex`  
풀스크린 높이 지정: `dvh`,`vh`
```
