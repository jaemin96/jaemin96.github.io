## 🤖 AI 에이전트와 함께 성장하기 - 의존이 아닌 협업의 기술

> Claude Code를 비롯한 AI 에이전트 도구들이 개발 생산성을 극적으로 높여주는 건 사실이다.  
> 하지만 사수 없는 환경에서 AI에 대한 의존도가 높아지면서, "이게 맞는 방향일까?"라는 고민이 생겼다.  
> 이 글은 AI 도구를 활용하면서도 개발자로서의 성장을 놓치지 않기 위한 나만의 가이드라인이다.

### 배경: 왜 이런 고민을 하게 됐나

현재 나는 4년차 FE 개발자로 React를 주로 다루고 있다. 신입부터 현재 회사까지 모두 따로 사수가 없어서 모르는 건 직접 찾아보거나 삽질하면서 배워왔다.  
AI 기술이 확산되기 이전에 개발자로 일을 시작했고, 항상 모르는 건 `stack overflow (최고였지..)`나 `youtube`, 강의사이트 (`udemy, inflearn`) 활용이 전부 였었다.

최근 Claude Code 같은 AI 에이전트 도구를 접하고 나서 생산성이 확실히 올라갔다. 막히는 부분을 바로바로 물어볼 수 있고, 반복적인 코드 작성도 빠르게 처리할 수 있었다.

그런데 문득 이런 생각이 들었다.

"에러 나면 AI한테 먼저 물어보는 게 습관이 되고 있는데, 이거 괜찮나?"

직접 디버깅하고 공식 문서 찾아보던 시간이 줄어들면서, 오히려 근본적인 이해도가 떨어지는 건 아닐까? 특히 사수가 없는 환경에서는 이런 부분을 체크해줄 사람도 없으니까.

그래서 AI 도구를 쓰되, "의존"이 아닌 "협업"의 관점에서 어떻게 활용할지 정리해봤다.

### AI 에이전트 사용의 양날의 검

#### ✅ 긍정적인 면

- **빠른 문제 해결**: 막힐 때 바로바로 물어보고 진행할 수 있어서 생산성 up
- **학습 속도 가속**: 새로운 기술 스택이나 패턴을 빠르게 시도해볼 수 있음
- **반복 작업 감소**: 보일러플레이트 코드 작성 시간을 줄이고 본질적인 로직에 집중 가능

#### ⚠️ 위험한 면

- **표면적 이해**: "왜 이렇게 동작하는지" 모르고 넘어가기 쉬움
- **디버깅 능력 저하**: 에러 나면 AI한테 물어보는 게 습관이 되면, 스스로 문제를 해결하는 능력이 안 늘어남
- **코드 퀄리티 판단 불가**: "AI가 짜준 거니까 괜찮겠지"가 기준이 되면 안티패턴을 모르고 넘어갈 수 있음

결국 핵심은 `"AI를 어떻게 활용하느냐"`다.

### 실전 가이드라인

#### 1. "왜?"를 3번은 물어보기

AI가 코드를 짜주면 그냥 복붙하고 끝내는 게 아니라

```m
1차: "왜 이렇게 했어?"
2차: "그럼 다른 방법은 없어?"
3차: "각 방법의 트레이드오프는 뭐야?"
```

이렇게 하면 단순히 "동작하는 코드"가 아니라 "이해한 코드"가 된다.

**예시:**

```typescript
// AI가 짜준 코드
const fetchData = async () => {
  try {
    const response = await fetch('/api/data');
    const data = await response.json();
    return data;
  } catch (error) {
    console.error(error);
  }
};
```

**내가 해야 할 질문들:**

- "왜 try-catch를 썼어? 에러 핸들링 다른 방법은?"
- "response.ok 체크는 안 해도 돼?"
- "에러를 throw하지 않고 console.error만 하면 어떤 문제가 있어?"

#### 2. AI 코드를 그대로 쓰지 말고 리팩토링해보기

claude code를 이용해보며 느낀 점은 방향을 제시하고 특정 UI를 추가해주는 과정에서 이게 맞나 싶을 정도로 기능을 정말 잘 구축해준다.  
하지만 나만의 스타일이 있으니 당연히 리펙토링이나 개선해야 할 점들이 명확히 보이긴 한다.

간단한 예를 들자면, React에서 한 컴포넌트에서 2~3번 사용되는 버튼이 있다면 이를 inline-style 코드 형태로 스타일 정의를 똑같이 3번 해버린다는 점.  
이러한 경우 요구사항을 추가로 줘서 컴포넌트로 빼라던가 스타일을 한번만 정의해서 다 반영해달라는 추가 요구를 통해 개선할 수야 있겠지만, 나는 이렇게 기본 골격만 잡아주면 그걸 리펙토링 하면서 어떤식으로 구축했는 지 살펴보고 내 입맛대로 다시 고쳐서 사용하는 편이다.

- 변수명을 내 컨벤션에 맞게 바꾸기
- 구조를 다시 설계해보기
- "이거 내가 짰다"고 자신있게 말할 수 있을 정도로 소화하기

이 과정을 거치면 단순히 복붙한 코드가 아니라 내 이해도가 향상되어 나의 작업이 된다고 느꼈다.  
(그냥 claude code가 전부 고쳐준 상태로 넘어가면 팀원한테 설명도 못하는 민망한 상황이 생긴적이 있었다ㅠ)

#### 3. 디버깅은 혼자 먼저 30분

에러가 나면 바로 AI한테 물어보지 말고 삽질하기

1. 콘솔 로그 찍어보기
2. 공식 문서 찾아보기
3. 에러 메시지 구글링해보기
4. 30분 삽질했는데도 안 되면 그때 AI 활용

**AI는 "힌트 주는 선배"지 "대신 해주는 외주"가 아니다.**

AI한테 바로 물어보면 "해결 방법"은 얻지만 "문제를 푸는 능력"은 안 늘어난다.

#### 4. 주기적으로 AI 없이 코딩하는 날 만들기

일주일에 하루는 AI 도구를 끄고 작업해보자.

- 내 실력을 체크하는 용도
- 막히는 부분 = 내가 진짜 모르는 부분
- 그 부분을 다시 공부하는 사이클

이게 없으면 "AI 없이는 못 만드는 개발자"가 될 수 있다.

#### 5. AI한테 코드 짜달라고 하지 말고, 설명 먼저 들어보기

질문 방식을 바꾸는 것만으로도 학습 효과가 달라진다:

**❌ 안 좋은 질문:**

```m
"React에서 무한 스크롤 구현해줘"
```

**✅ 좋은 질문:**

```m
"React에서 무한 스크롤을 구현하려면 어떤 접근 방법이 있을까?
각 방법의 장단점은 뭐야?"
```

→ 방향을 잡고 → 내가 직접 짜보고 → 막히면 그때 구체적으로 도움 요청

이렇게 하면 단순히 "코드 복붙"이 아니라 "문제 해결 능력"이 늘어난다.

#### 6. 코드리뷰를 AI한테 받기

내가 짠 코드를 AI한테 보여주고

- "이 코드 개선할 점 있어?"
- "이 부분 왜 이렇게 짰는지 설명해줄게, 더 나은 방법 있어?"
- "이거 프로덕션 레벨에서 쓰기에 괜찮아?"

**사수가 없는 환경**에서 이게 진짜 중요하다. **안티패턴**을 혼자서는 모를 수 있으니까.

### 사수 없는 환경에서 특히 조심할 점

#### 1. 안티패턴을 모르고 넘어가기 쉬움

AI가 짜준 코드가 "동작은 하는데" 실무에선 쓰면 안 되는 방식일 수 있다.

**예시:**

```typescript
/* 동작은 하지만 안티패턴 */
const [data, setData] = useState([]);

useEffect(() => {
  fetchData().then(result => setData(result));
}, []);  // dependency 누락
```

주기적으로 "이 코드 프로덕션에 써도 괜찮아?" 물어보고, 왜 안 되는지 이해하는 시간이 필요하다고 판단.

#### 2. 컨벤션/베스트 프랙티스를 몰라도 티가 안 남

**혼자 개발하다 보면 내 코드가 업계 표준과 얼마나 다른지 모를 수 있다.**  
이게 참 딜레마인 것 같다... 아키텍처나 다른 코드들을 참고해서 리펙토링을 해도 그게 맞는 건지 오버엔지니어링인 지 알수가 없다.

**해결책:**

- 회사 코드 스타일 가이드가 있으면 그걸 기준으로 (이건 현재 없고 프론트도 내가 리드인 웃픈 상황, 내 스타일이 곧 가이드라인)
- "React 커뮤니티에서 권장하는 방식은 뭐야?" 같은 질문 자주 하기
- 오픈소스 잘 만들어진 프로젝트 코드 읽어보기

#### 3. 트러블슈팅 경험이 안 쌓임

에러를 만났을 때가 삽질하면서 생각도 가장 많이 하고 이런저런 시도도 가장 많이하는 시간인데, AI한테 바로 물어보면서 그런 경험들이 줄어들고 있다.

**중요한 건:**

- 에러 로그 읽는 법
- 공식 문서 찾는 법
- 스택 트레이스 따라가는 법

이런 게 진짜 실력이고, AI는 이걸 대신해줄 수 없다.

### 실제 적용 사례

최근에 내가 진행 중인 CI/CD 학습에도 같은 원칙을 적용했다.

#### 잘못된 접근 (초기)

```m
"Claude Code야, GitHub Actions로 배포 자동화 파이프라인 짜줘"
```

→ yml 파일은 받았지만 내가 기존 관리하던 flow에서는 없던 단계들이 신규로 추가되어도 해당 단계들이 어떤 역할을 하길래 추가 된건지 필요성을 모름.

#### 개선된 접근 (현재)

```m
"GitHub Actions로 React 앱을 자동 배포하려면 
어떤 단계들이 필요해? 각 단계의 역할은 뭐야?"
```

→ 전체 플로우 이해 → 각 단계 직접 작성 → 막히면 구체적으로 질문

결과적으로:

- yml 파일을 이해하면서 작성할 수 있게 됨
- 당장 필요한 단계인 지 생략가능한 단계인 지 파악 가능
- 나중에 다른 프로젝트에도 응용 가능
- 에러 나도 어디서 문제인지 파악 가능

### 마무리

AI 에이전트 도구는 분명 강력하다. 생산성도 올라가고 학습 속도도 빨라지며 다방면으로 장점이 더 많은 것은 팩트이다.

하지만 핵심은 **AI를 "어떻게" 활용하느냐**다.

예를 들자면 비서가 사장 대신 서류에 서명이나 승인을 하진 않는 것처럼 결국 결정권자는 나 본인이 되어야한다. 그러한 결정을 하기 위해서는 당연히 결정할 사항에 대한 이해도가 바탕이 되어야한다.

- 빠른 해결보다 깊은 이해가 장기적으로 더 중요
- 사수 없는 환경이라면 더더욱 스스로 체크하는 습관이 필요

이 가이드라인을 따르면:

1. AI 도구의 생산성 이점은 다 가져가면서
2. 실력은 계속 늘고
3. 나중에 AI 없어도 혼자 할 수 있는 능력 유지

결국 AI는 도구일 뿐이고, 그 도구를 어떻게 쓰느냐는 개발자인 우리의 몫이다.

**핵심 체크리스트**:

- ✔ AI 코드 **3번** 더 생각하기
- ✔ 에러에 직면하면 디버깅 후 AI의 도움 받기
- ✔ AI 없이도 코딩하는 시간 가지기
- ✔ 질문 방식 바꾸기: "짜줘" → "어떻게 접근할까?" (Claude code - write 기능보단 plan 기능 활용)
- ✔ 내가 짠 코드를 AI한테 리뷰 받기
- ✔ 마지막엔 프로덕션 레벨 코드인지 항상 체크
