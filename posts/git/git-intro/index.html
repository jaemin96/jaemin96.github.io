나는 Git을 올바르게 쓰고 있었나?

20 Apr, 2026
| Edit page

들어가며

사실 Git의 중요성과 학습 필요성은 진작 인지하고 있었으며, 내용 정리도 진행했었다.
그러나, 딱 거기까지였고 그 이상으로 뭔가 이뤄진 것은 없는 상태였다.

항상 쓰는 명령어들만 쓰거나 조금 복잡하거나 번거로운 경우는 GUI에 의존하려 하고
GUI에서 버튼 클릭을 통해 처리된 것들을 CLI 환경에서 똑같이 처리하려 하면 난항을 겪었다.

이런 상황이 반복되고 Git의 중요성에 대해 간헐적으로 인지할때마다 생각만 하고 끝나고
더이상의 발전이 없다고 판단하여

아, 이건 아닌 것 같다. 명확히 파악하고 정리해서 체계적으로 진행하자

라는 결과에 도달하였고, 이 글이 그 시작점이다.

근데 Git은 왜 쓰는 것이며, 정확하게 알고 쓰고 있는가?

부끄럽지만 누군가 나에게 이런 질문을 던진다면,

Git은 형상관리 도구라서 코드 관리가 용이하며,
작업한 내역은 add 명령어로 스테이징 단계로 올려두고 검증 후 commit 및 push 합니다.
작업중인데 완료되지 않고 브랜치를 변경하거나 환경을 잠깐 바꿔야할때는 stash & pop 으로 임시저장 및 불러오기를 이용합니다.
그리고 원격저장소에 신규 업데이트 된 내역이 있다면 pull을 받습니다.

답변 자체는 틀리지 않았다고 자신하지만, 좀 더 디테일하게 대답을 요구한다면 쉽게 이 이상을 설명할 순 없을 것이다.

그래서, Git이 정확히 무엇일까?

일단 ‘형상관리’ 라는 말부터 이해할 필요가 있다

형상관리란 소프트웨어 개발 과정에서 발생하는 모든 변경 사항들을 체계적으로 추적하고 통제하는 활동을 말한다.
여기서 말하는 모든 변경 사항들에는 코드 변경, 문서, 라이브러리의 변경 같은 내용들을 포함한다.

Git을 통해 형상관리하는 플로우에 대하여 알아보자

  Working Directory → [git add]  
  → Staging Area → [git commit]  
  → Local Repository → [git push]  
  → Remote Repository
  • Working Directory

현재 본인이 작업하고 있는 공간으로 파일 추가나 수정 모두 여기에 해당한다.

단, Git이 변경사항들을 감지하도록 작업 디렉토리를 git 관리 대상으로 만들어줘야한다.
git init 명령으로 디렉토리 경로에 .git 폴더를 만들어주면 git이 디렉토리를 추적하기 시작한다.

  • Staging Area

git add 명령을 통해 올라온 변경사항들이 대기하는 공간을 뜻한다.

Commit 할때 이 변경사항들은 모두 포함할게

라는 의사 표시 단계이며, 커밋전에 무엇을 기록할 지 선택이 가능하다는게 git의 특징 중 하나이다.

  • Local Repository

git commit 명령을 통해 Staging Area에 포함시킨 내용들의 스냅샷을 기록하는 곳이다.
이 시점부터 변경사항이 이력으로 기록되며, 로컬 변경사항은 인터넷 없이도 관리할 수 있다.

  • Remote Repository

git push 명령을 통해 로컬에서만 관리하던 변경사항을 원격저장소에 업데이트 해줄 수 있다.
반대로 git pull 명령을 통해 원격저장소에 변경사항을 로컬에 가져올 수도 있다.

내가 Git을 쓰면서 자주 쓰는 것들

내가 Git을 활용하면서 개발할 때 가장 많이 쓰는 명령이 몇 가지 있다.

git status
이 명령은 지금까지 내가 어떤 작업들을 했는지, 어떤 작업들이 스테이징 영역에 올라가 있는지 목록들을 표시해준다.

git add . 이 명령은 지금까지 작업한 모든 사항들을 전부 스테이징 영역으로 옮겨준다. 여기서 .은 현재 디렉토리를 의미한다.
내가 src 경로에 위치한 상태라면 src 내에서 작업한 변경사항들은 모두 스테이징으로 이동되는 것이다.

어느 정도 명령어가 익숙해지면 alias를 등록해서 사용할 수 있다.

숏컷 설정법은 여기서는 다루지 않겠다. 그리고 명령어가 익숙치 않은 초기에는 숏컷을 사용하는 것은 추천하지 않는다.
걷지도 못하는데 뛰는것을 가르치지 않는 것처럼 확실하게 기본부터 배우는게 맞다.

git add . >> git aa  
git status >> git st  
git commit >> git cm  
git push >> git ps  
git pull >> git pl

본격적으로 방향성을 설정하자

앞서 말했듯이 Git의 기본 흐름은 어느 정도 파악하고 있지만, 항상 같은 명령어만 반복하고, 조금만 복잡해지면 GUI에 의존한다는 문제가 명확하다.

앞으로 아래 순서로 학습을 진행하려 한다.

1단계 - Git 내부 동작 이해

단순히 명령어를 외우는 것이 아니라,
Git이 내부적으로 어떻게 동작하는지를 먼저 이해가 필요하다.
.git 폴더 구조, object, HEAD, index 파일이 실제로 무엇인지 파악하면 명령어 하나하나가 어떤 의미인지 자연스럽게 이해된다.

2단계 - 브랜치 전략

브랜치의 생성/삭제/이동이 내부적으로 어떻게 이루어지는지 이해하고,
merge와 rebase의 차이, 그리고 어떤 상황에서 무엇을 써야 하는지를 정리한다.

3단계 - 협업

PR 리뷰 플로우, conflict 해결, cherry-pick / reset / revert 의 차이와 올바른 사용 시점을 다룬다.

4단계 - 실무

Git Flow, GitHub Flow 같은 브랜치 전략 패턴과 commit message 컨벤션과
.gitignore, submodule 같은 실무에서 자주 마주치는 내용들을 정리한다.
